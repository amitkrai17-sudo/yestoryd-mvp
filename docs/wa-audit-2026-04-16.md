## Phase 0 — Communication library surface

### `lib/communication/aisensy.ts`
- **Exports:**
  - `sendWhatsAppMessage(params)` — single AiSensy outbound function. Logs every send (success/failure) to `communication_logs` via `logCommunication()`.
  - `isWhatsAppConfigured()` — boolean helper.
  - Types: `AiSensyButton`, `AiSensySendMeta`.
- Calls `fetch` against `AISENSY_BASE_URL` exactly once, inside this file.

### `lib/communication/index.ts`
- **Exports:**
  - `sendCommunication(params)` — template-driven send: loads row from `communication_templates`, respects parent prefs, calls `sendWhatsAppToRecipient()` (which delegates to `sendWhatsAppMessage`) and/or `sendEmail()`.
  - `scheduleCommunication(params)` — enqueues to `communication_queue`.
  - `sendAdminAlert(templateCode, vars, ...)` — thin wrapper over `sendCommunication`.
  - Re-exports: `sendWhatsAppCloudMessage`, `markMessageAsRead`, `isWhatsAppCloudConfigured`, `logCommunication`.
- Types: `SendCommunicationParams`, `SendCommunicationResult`.

### `lib/communication/log.ts`
- **Exports:**
  - `logCommunication(params)` — inserts into `communication_logs` (boolean wa_sent/email_sent/sms_sent, flat recipient fields, extras in `context_data` JSONB).
  - Types: `RecipientType`, `TriggeredBy`, `LogCommunicationParams`.

**Single outbound function?** YES — `sendWhatsAppMessage()` is the only place `fetch(backend.aisensy.com)` is intended to execute. One bypass exists (see Phase 1).

## Key findings

1. **One true send-path bypass:** `app/api/auth/send-otp/route.ts` issues its own `fetch` against AiSensy. Worth migrating to `sendWhatsAppMessage({ templateName: 'parent_otp_v3', buttons: [...], meta: {...} })` — it already supports buttons.
2. **One DB flag mismatch:** `C9_session_reminder` row has `use_whatsapp=false` but code actively sends `coach_session_reminder_1h_v3` via direct `sendWhatsAppMessage` (bypasses the `sendCommunication` flag-check). Either flip the flag to `true` or migrate the caller to `sendCommunication`.
3. **CODE ONLY templates (~28):** Most admin alerts, tuition, group-class, OTP, proactive notifications, and goals-capture templates are NOT represented in `communication_templates`. They're hardcoded campaign-name strings sent directly via `sendWhatsAppMessage`. This is consistent (two send paths: DB-driven `sendCommunication` vs. hardcoded `sendWhatsAppMessage`), but means those templates bypass parent notification preferences, email fallback, and the admin-managed template editor.
4. **DB ONLY templates (~26):** Fully managed through `sendCommunication(templateCode)`. No orphans — each has at least one caller using its `template_code`.
5. **Slug-based lookups** (`discovery-call/[id]/send-payment-link`, `send-followup`) read templates by `slug` column, which is a separate indirection worth auditing for consistency with `template_code`.
